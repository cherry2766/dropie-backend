# 동시성 테스트 측정 결과

## 측정 환경
- **테스트 코드**:
  - `OptimisticLockConcurrencyTest.낙관적_락_15회_반복_측정`
  - `RedisLockConcurrencyTest.Redis_분산_락_15회_반복_측정`
- **DB**: H2 in-memory (테스트 환경)
- **동시 주문 수**: 100 스레드 (`Executors.newFixedThreadPool(100)`)
- **초기 재고**: 50개
- **반복 횟수**: 15회 (각 회차마다 DB 리셋 + 동일 조건 재현)
- **측정 일자**: 2026-05-07

## 측정 의도
충돌 빈도가 높은 선착순 주문 환경에서 락 전략별 정합성·처리량(TPS)·응답 시간(p95) 트레이드오프를 정량 비교하기 위함.
- 낙관적 락(JPA `@Version`) + `@Retryable(maxAttempts=3, backoff 100→200→400ms)` + `@Recover`
- Redis 분산 락(Redisson `tryLock(waitTime=5s, leaseTime=3s)`)

## 핵심 인사이트
- **데이터 정합성**: 두 전략 모두 음수 재고 0건, 정합성 100% 보장.
- **미판매율**: 낙관적 락 24% vs 분산락 0% — 충돌이 3회 재시도 안에 풀리지 않아 `ORDER_CONFLICT(409)`로 응답되는 케이스가 많음.
- **TPS는 비슷**: 평균 처리량 약 100 TPS로 두 전략 거의 동등.
- **p95는 분산락이 더 빠름** (의외): 일반적으로 "낙관적 락이 빠르다"는 통념이 있지만, **고경쟁 환경에서는 재시도 백오프(100→200→400ms)가 누적되어 꼬리 응답이 더 길어진다**. 분산락은 직렬화로 p50은 살짝 느리지만 백오프 없이 처리되어 p95가 18% 빠름.
- **결론**: 선착순 드롭처럼 충돌이 폭증하는 구간에서는 **분산락이 정합성 + 꼬리 응답 모두 우위**. 낙관적 락은 충돌 빈도가 낮은 일반 주문에 적합. `app.lock.type` 프로퍼티 토글로 운영 중 전환 가능한 구조.

## 재현 방법
```bash
# 낙관적 락 측정
./gradlew test --tests "OptimisticLockConcurrencyTest.낙관적_락_15회_반복_측정"

# 분산락 측정 (Redis 실행 필요: docker-compose up redis)
./gradlew test --tests "RedisLockConcurrencyTest.Redis_분산_락_15회_반복_측정"
```
각 테스트 한 번 실행 시 내부에서 15회 반복 후 평균/최소/최대 자동 출력.
