1# 동시성 테스트 측정 결과

## 측정 환경
- **테스트 코드**:
  - `OptimisticLockConcurrencyTest.낙관적_락_15회_반복_측정`
  - `RedisLockConcurrencyTest.Redis_분산_락_15회_반복_측정`
- **DB**: H2 in-memory (테스트 환경)
- **동시 주문 수**: 100 스레드 (`Executors.newFixedThreadPool(100)`)
- **초기 재고**: 50개
- **반복 횟수**: 15회 (각 회차마다 DB 리셋 + 동일 조건 재현)
- **측정 일자**: 2026-05-07

## 측정 의도
충돌 빈도가 높은 선착순 주문 환경에서 락 전략별 처리량과 미판매 발생률을 비교하기 위함.
- 낙관적 락(JPA `@Version`) + `@Retryable(maxAttempts=3)` + `@Recover` 조합
- Redis 분산 락(Redisson `tryLock(waitTime=5s, leaseTime=3s)`)

| 음수 재고 | 0건 | 0건 |

## 결론
- **데이터 정합성**: 두 전략 모두 음수 재고 0건, 정합성 100% 보장.
- **낙관적 락 한계**: 100스레드 고경쟁 환경에서 평균 12개(24%) 미판매. `@Version` 충돌이 3회 재시도 안에 풀리지 않아 `ORDER_CONFLICT(409)`로 응답.
- **분산락 우위**: 동일 조건에서 정확히 50건 성공, 락 획득 실패 0건. 충돌 빈도 높은 구간에서는 분산락이 명확히 유리.
- **선택 기준**: 충돌 빈도가 낮은 일반 주문은 낙관적 락(DB 부하 적음), 선착순 드롭처럼 충돌이 폭증하는 구간은 분산락. `app.lock.type` 프로퍼티 토글로 운영 중 전환 가능.

## 재현 방법
```bash
# 낙관적 락 측정
./gradlew test --tests "OptimisticLockConcurrencyTest.낙관적_락_15회_반복_측정"

# 분산락 측정 (Redis 실행 필요: docker-compose up redis)
./gradlew test --tests "RedisLockConcurrencyTest.Redis_분산_락_15회_반복_측정"
```
각 테스트 한 번 실행 시 내부에서 15회 반복 후 평균/최소/최대 자동 출력.
